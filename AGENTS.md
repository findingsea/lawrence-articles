# Lawrence 的文章库 (lawrence-articles)

## 项目说明

本文档库用于存放书籍读后感与评价类文章，辅助 AI 生成符合 Lawrence 写作风格的草稿。

## 目录结构

```
lawrence-articles/
├── AGENTS.md          # 本文件：项目规范与写作指南
├── post/              # 文章目录（含已发布和进行中的文章）
│   └── YYYY-MM-文章主题/            # 每篇文章一个文件夹
│       ├── 文章主题.md               # 主文章
│       ├── memo.md                  # 调研笔记、讨论记录
│       └── image/                   # 图片素材
└── image/             # 根级图片资源（较少用）
```

**说明**：不单独设 `draft/` 目录，所有文章（含进行中和已发布的）都放在 `post/` 下，用 Git 分支区分状态。写新文章时从 main 新建 feature 分支，merge 后即发布。

## 命名规范

### 文件夹命名
- **格式**：`YYYY-MM-文章主题/`
- **示例**：`2026-04-ai-era-engineer-adaptation/`
- **规则**：
  - 年份-月份开头（4位年份 + 2位月份）
  - 使用英文小写，单词用连字符分隔
  - 简洁明了，能概括文章核心内容

### 文件命名

| 文件类型 | 命名格式 | 示例 |
|---------|---------|------|
| 主文章 | `post/YYYY-MM-主题/文章主题.md` | `post/2026-04-ai-era-engineer-adaptation/ai-era-engineer-adaptation.md` |
| memo 文件 | `post/YYYY-MM-主题/memo.md` | `post/2026-04-ai-era-engineer-adaptation/memo.md` |

**注意**：
- 文章文件**与文件夹同名**（去掉日期前缀）
- memo 文件**保持原名** `memo.md`，用于记录讨论和素材
- 图片放在 `post/YYYY-MM-主题/image/` 下

## 协作写作流程（已验证版本）

```
讨论 → 写入 memo.md → AI 生成草稿 → 用户大幅修改 → 迭代完善 → merge 发布
         ↑___________________________________↓
```

**实际协作模式**（来自 2026-04 AI 时代生存思考文章写作验证）：

1. **讨论阶段**：用户与 AI 就文章主题进行讨论（飞书消息来回）
2. **记录到 memo**：AI 将讨论内容、调研摘要、参考资料链接整理到 `memo.md`
3. **生成草稿**：AI 基于 memo 生成文章草稿，写入 `主题.md`
4. **用户修改**：用户在本地修改草稿，幅度通常很大（删 30-50% 内容是常态）
5. **迭代讨论**：用户发回修改稿 → AI 从修改中学习规则 → 下一轮写得更准
6. **Git 协作**：每轮修改 commit push，用户 merge 到 main

**关键特点**：
- 用户给的是执行型反馈（不是「这里不好」，而是「改成这样」），很高效
- AI 从用户删改中学到写作风格，修改量逐轮减少
- 用户有很强的写作风格感，会主动删 AI 味的表述
- 数据必须有来源，无来源宁可删掉不用

### Git 分支使用规范

```
feature/YYYY-MM-文章主题   # 当前文章的分支
main                        # 已发布的正式内容
```

- 每次 push 到 feature 分支，用户 merge 到 main
- commit message 用 `feat:` `fix:` `refactor:` `style:` 等前缀

## memo 文件用途

`draft/YYYY-MM-主题/memo.md` 是**讨论记录和素材汇总**，包含：
- 行业趋势调研
- 核心问题讨论（Q&A 形式）
- 参考资料链接
- 灵感/想法记录
- 待办事项

**注意**：memo 是工作文件，不是最终文章。正式文章基于 memo 生成。

## 文章结构规范

### 深度书评（单本书）

1. **前言**
   - 交代读这本书的契机或最初的感触
   - 可以引用在朋友圈/豆瓣发过的短评
   - 或直接点出这本书最打动你的点

2. **正文（分点展开）**
   - 每个小节围绕一个核心观点展开
   - 结合书的内容，但不能止于情节复述
   - 重点分析**写法、形式、技巧**（叙事手法、人物塑造、矛盾设置、结构编排等）
   - 善用对比和关联（关联其他书籍、电影、作品）
   - 可以引用原文让作者观点自己说话

3. **结语**
   - 总结性评价或推荐语
   - 不要空洞，要有具体指向

### 月度阅读报告（多本书）

- 每本书：豆瓣链接 + 封面图 + 简短评价（200-500字）
- 评价精准，一针见血
- 不面面俱到，只讲最突出的点

### 技术/观点类文章

- **前言**：交代背景和写作动机
- **正文**：分点论述，逻辑清晰
- **结语**：总结观点或给出建议

## 行文风格（根据历史文章总结）

### 核心原则

**1. 个人视角鲜明**
- 大量使用"我觉得"、"我认为"、"我想尝试讲的是"
- 不回避个人感受和偏好
- 敢于表达负面评价（"推荐度比较低"、"有点令人失望"）

**2. 不剧透，注重分析（书评适用）**
- 明确说："故事内容本身不用多讲，一来是剧透了再读的乐趣就少得多"
- 重点分析**写法、形式、技巧**（叙事手法、人物塑造、矛盾设置）
- 跳脱内容猎奇，关注"作者怎么把故事讲得令人手不释卷"

**3. 善用对比和关联**
- 通过关联其他作品/案例突出主题
- 阅读量大，信手拈来
- 常见参照：金庸、《冰与火之歌》、《阿甘正传》、《哈利波特》等

**4. 有深度但不高冷**
- 用通俗语言讲清楚观点
- 不故弄玄虚，不掉书袋
- "写的非常浅显易懂，即便是文科生看懂整个逻辑也是没有任何问题"

**5. 犀利直接**
- 敢说真话，不人云亦云
- "舒尔茨明显不是一个好作家"
- "刘瑜到底也没带来新的惊喜"
- "口味彻底被养叼了"

### 语言特征

- **干净利落**：无 emoji，纯文字
- **中文为主**：英文术语自然融入
- **篇幅灵活**：短评几百字，深度文章可达上万字
- **善用引用**：引用原文/数据让观点更有说服力
- **豆瓣链接（书评）**：书名必带豆瓣链接，阅读报告配封面图
- **引号使用**：使用「」而非""，例如「古法编程」「技能匹配」

### 写作风格要点（从文章协作中提炼）

> ⚠️ **每次写文章前，请完整阅读本节所有内容。禁止跳过或凭记忆写。**

**像跟朋友聊天，不是写论文或做报告。**

1. **去元叙述**：直接说观点，不要铺垫「这是我的观察」「这里我比较确定」「我的判断是」。观点直接说，不用告诉读者「我要告诉你」。

2. **更口语化自然**：句子短，读起来像在跟人聊天。用「我觉得」「我认为」但不要过度。不用过于正式的书面语。

3. **人物介绍**：直接介绍人名和职务，不要为每个人物加一句评价性引导语。❌ 「**OpenAI 内部已经在发生。** OpenAI API 及开发者平台工程负责人 Sherwin Wu 透露……」→ ✅ 「Sherwin Wu，OpenAI API 及开发者平台工程负责人，透露说……」

4. **不居高临下**：避免「赢家」「边缘化」「淘汰的信号」这类审判式措辞。用更温和但同样有力量的表达。

5. **用个人经验增加说服力**：加入共情和自己的感受，而不是纯观察视角。❌ 「技术名词迭代极快」→ ✅ 「很多人会觉得我 Context Engineering 还没学会呢，怎么又出了个 Harness 是什么？」可以加「也是我自己的学习路径」这类表述。

6. **列表 vs 自然段落**：简单并列可以用列表，但需要解释的观点用自然段落更好。不要为了结构工整而强行列表。

7. **措辞精确克制**：不要用过于强烈的修饰词。❌ 「重新定义」「独特挑战」「角色也在扩展」→ ✅ 「重新明确」「额外挑战」「工作更重」

8. **敢于删减**：如果一个小节对核心论点帮助不大，果断删掉。不要为了面面俱到而留冗余内容。结尾不要逐条复述正文，简洁有力即可。

9. **标题简洁**：不要过于戏剧化或学术化。❌ 「一场正在发生的大洗牌」「转型策略」「最值得投入的新能力」→ ✅ 「一场正在发生的变革」「转型思考」「AI 学习优先级」

## 禁止事项

- 不做流水账式的情节复述
- 不堆砌摘抄原文
- 不写空洞的"这本书很好，推荐大家读"类结语
- 不故弄玄虚、掉书袋
- 不人云亦云，没有自己的观点
- 不写元叙述（「这是我的观察」「这里我比较确定」等）
- 不居高临下（避免「赢家」「边缘化」等词）
