# Lawrence 的文章库 (lawrence-articles)

## 项目说明

本文档库用于存放书籍读后感与评价类文章，辅助 AI 生成符合 Lawrence 写作风格的草稿。

## 目录结构

```
lawrence-articles/
├── AGENTS.md          # 本文件：项目规范与写作指南
├── posts/              # 文章目录（含已发布和进行中的文章）
│   ├── index.md       # 文章索引：按主题分类，供 AI 快速定位文章
│   └── YYYY-MM-文章主题/            # 每篇文章一个文件夹
│       ├── 文章主题.md               # 主文章
│       ├── memo.md                  # 调研笔记、讨论记录
│       └── images/                   # 图片素材
└── images/             # 根级图片资源（较少用）
```

**说明**：不单独设 `draft/` 目录，所有文章（含进行中和已发布的）都放在 `posts/` 下，用 Git 分支区分状态。写新文章时从 main 新建 feature 分支，merge 后即发布。

### 文章索引（posts/index.md）

`posts/index.md` 是所有文章的分类索引，供 AI 快速定位相关文章，避免每次遍历全部文件。

**用途**：
- AI 需要检索过往文章时，先读 `posts/index.md`，根据总览表格定位到相关分类，再只读对应文章
- 写新文章时，AI 可以通过索引发现相关主题的过往文章，保持观点连贯

**维护规则**：
- 每篇新文章 merge 到 main 后，在 index.md 对应分类下追加一条
- 条目格式：`- [文章标题](相对路径) — 一句话摘要`
- 摘要控制在30字以内，突出文章核心内容
- 如果现有分类不合适，可以新建分类，但优先复用已有分类

## 命名规范

### 文件夹命名
- **格式**：`YYYY-MM-文章主题/`
- **示例**：`2026-04-你记不住读过的书/`
- **规则**：
  - 年份-月份开头（4位年份 + 2位月份）
  - 使用中文命名
  - 简洁明了，能概括文章核心内容

### 文件命名

| 文件类型 | 命名格式 | 示例 |
|---------|---------|------|
| 主文章 | `posts/YYYY-MM-主题/文章主题.md` | `posts/2026-04-你记不住读过的书/你记不住读过的书.md` |
| memo 文件 | `posts/YYYY-MM-主题/memo.md` | `posts/2026-04-你记不住读过的书/memo.md` |

**注意**：
- 文章文件**与文件夹同名**（去掉日期前缀）
- memo 文件**保持原名** `memo.md`，用于记录讨论和素材
- 图片放在 `posts/YYYY-MM-主题/images/` 下

## 协作写作流程（已验证版本）

```
讨论 → 写入 memo.md → AI 生成草稿 → 用户大幅修改 → 迭代完善 → merge 发布
         ↑___________________________________↓
```

**实际协作模式**（来自 2026-04 AI 时代生存思考文章写作验证）：

1. **讨论阶段**：用户与 AI 就文章主题进行讨论（飞书消息来回）
2. **检索过往文章**：AI 先读 `posts/index.md` 定位相关文章，再只读对应文章了解已有观点
3. **记录到 memo**：AI 将讨论内容、调研摘要、参考资料链接整理到 `memo.md`
4. **生成草稿**：AI 基于 memo 生成文章草稿，写入 `主题.md`
5. **用户修改**：用户在本地修改草稿，幅度通常很大（删 30-50% 内容是常态）
6. **迭代讨论**：用户发回修改稿 → AI 从修改中学习规则 → 下一轮写得更准
7. **Git 协作**：每轮修改 commit push，用户 merge 到 main
8. **更新索引**：文章发布后，在 `posts/index.md` 对应分类下追加条目

**关键特点**：
- 用户给的是执行型反馈（不是「这里不好」，而是「改成这样」），很高效
- AI 从用户删改中学到写作风格，修改量逐轮减少
- 用户有很强的写作风格感，会主动删 AI 味的表述
- 数据必须有来源，无来源宁可删掉不用

### Git 分支使用规范

```
feature/YYYY-MM-文章主题   # 当前文章的分支
main                        # 已发布的正式内容
```

- 每次 push 到 feature 分支，用户 merge 到 main
- commit message 用 `feat:` `fix:` `refactor:` `style:` 等前缀

## memo 文件用途

`posts/YYYY-MM-主题/memo.md` 是**讨论记录和素材汇总**，包含：
- 行业趋势调研
- 核心问题讨论（Q&A 形式）
- 参考资料链接
- 灵感/想法记录
- 待办事项

**注意**：memo 是工作文件，不是最终文章。正式文章基于 memo 生成。

## 文章结构规范

### 深度书评（单本书）

1. **前言**
   - 交代读这本书的契机或最初的感触
   - 可以引用在朋友圈/豆瓣发过的短评
   - 或直接点出这本书最打动你的点

2. **正文（分点展开）**
   - 每个小节围绕一个核心观点展开
   - 结合书的内容，但不能止于情节复述
   - 重点分析**写法、形式、技巧**（叙事手法、人物塑造、矛盾设置、结构编排等）
   - 善用对比和关联（关联其他书籍、电影、作品）
   - 可以引用原文让作者观点自己说话

3. **结语**
   - 总结性评价或推荐语
   - 不要空洞，要有具体指向

### 月度阅读报告（多本书）

- 每本书：豆瓣链接 + 封面图 + 简短评价（200-500字）
- 评价精准，一针见血
- 不面面俱到，只讲最突出的点

### 技术/观点类文章

- **前言**：交代背景和写作动机
- **正文**：分点论述，逻辑清晰
- **结语**：总结观点或给出建议

## 行文风格（根据历史文章总结）

### 核心原则

**1. 个人视角鲜明**
- 大量使用"我觉得"、"我认为"、"我想尝试讲的是"
- 不回避个人感受和偏好
- 敢于表达负面评价（"推荐度比较低"、"有点令人失望"）

**2. 不剧透，注重分析（书评适用）**
- 明确说："故事内容本身不用多讲，一来是剧透了再读的乐趣就少得多"
- 重点分析**写法、形式、技巧**（叙事手法、人物塑造、矛盾设置）
- 跳脱内容猎奇，关注"作者怎么把故事讲得令人手不释卷"

**3. 善用对比和关联**
- 通过关联其他作品/案例突出主题
- 阅读量大，信手拈来
- 常见参照：金庸、《冰与火之歌》、《阿甘正传》、《哈利波特》等

**4. 有深度但不高冷**
- 用通俗语言讲清楚观点
- 不故弄玄虚，不掉书袋
- "写的非常浅显易懂，即便是文科生看懂整个逻辑也是没有任何问题"

**5. 犀利直接**
- 敢说真话，不人云亦云
- "舒尔茨明显不是一个好作家"
- "刘瑜到底也没带来新的惊喜"
- "口味彻底被养叼了"

### 语言特征

- **干净利落**：无 emoji，纯文字
- **中文为主**：英文术语自然融入
- **篇幅灵活**：短评几百字，深度文章可达上万字
- **善用引用**：引用原文/数据让观点更有说服力
- **豆瓣链接（书评）**：书名必带豆瓣链接，阅读报告配封面图
- **引号使用**：使用「」而非""，例如「古法编程」「技能匹配」

### 写作风格要点（从文章协作中提炼）

> ⚠️ **每次写文章前，请完整阅读本节所有内容。禁止跳过或凭记忆写。**

**像跟朋友聊天，不是写论文或做报告。**

1. **去元叙述**：直接说观点，不要铺垫「这是我的观察」「这里我比较确定」「我的判断是」「刘勃用……指出」「但某某认为」。观点直接说，不介绍「谁在说什么」，直接说事本身。

2. **更口语化自然**：句子短，读起来像在跟人说话。用「我觉得」「我认为」但不要过度。不用过于正式的书面语。❌ 「在我脑子里只是一个词」→ ✅ 「之前我对……的模糊印象就是」

3. **形容词克制**：修饰词越短越好。❌ 「这么多没被记住的东西」→ ✅ 「还有太多东西」

4. **人物介绍**：直接介绍人名和职务，不要为每个人物加一句评价性引导语。❌ 「**OpenAI 内部已经在发生。** OpenAI API 及开发者平台工程负责人 Sherwin Wu 透露……」→ ✅ 「Sherwin Wu，OpenAI API 及开发者平台工程负责人，透露说……」

5. **不居高临下**：避免「赢家」「边缘化」「淘汰的信号」这类审判式措辞。用更温和但同样有力量的表达。

6. **不给作者写评价**：不吹不踩，客观描述书是什么类型。❌ 「刘勃的价值在于」→ ✅ 「属于不会直接说答案的书」

7. **用个人经验增加说服力**：加入共情和自己的感受，而不是纯观察视角。❌ 「技术名词迭代极快」→ ✅ 「很多人会觉得我 Context Engineering 还没学会呢，怎么又出了个 Harness 是什么？」可以加「也是我自己的学习路径」这类表述。

8. **列表 vs 自然段落**：简单并列可以用列表，但需要解释的观点用自然段落更好。不要为了结构工整而强行列表。

9. **措辞精确克制**：不要用过于强烈的修饰词。❌ 「重新定义」「独特挑战」「角色也在扩展」→ ✅ 「重新明确」「额外挑战」「工作更重」。

10. **敢于删减**：如果一个小节对核心论点帮助不大，果断删掉。不要为了面面俱到而留冗余内容。结尾不要逐条复述正文，简洁有力即可。

11. **标题简洁**：不要过于戏剧化或学术化。❌ 「一场正在发生的大洗牌」「转型策略」「最值得投入的新能力」→ ✅ 「一场正在发生的变革」「转型思考」「AI 学习优先级」

12. **开头要有问题意识**：用疑问或张力牵引读者，不要平铺直叙。

13. **「有趣」优先于「有用」**：读后感的落脚点是阅读体验，不是功能价值。❌ 「读历史最有用的地方」→ ✅ 「读历史最有趣的部分」。

14. **用已有概念，不造新的描述**：❌ 「秦朝只撑了十四年」→ ✅ 「秦朝二世而亡」——读者已经知道的概念直接用。

15. **读后感核心是「我的阅读体验」，不是「书的总结」**：增写私人阅读感受。❌ 「这本书很好，推荐大家读」→ ✅ 「我自己阅读的感受，是比较轻松的……」或「它最吸引我的点是……」

16. **删评价语**，不需要对批评声音做回应：❌ 「这些声音我理解，但不影响我对它的评价」→ 直接删掉，读者自有判断。

17. **标题升级**：从书名→更能概括核心主题的标题。❌ 「希腊别传」→ ✅ 「希腊文化的生命力」——聚焦核心主线。

18. **中西方对比是丰富维度的好方法**：但确保事实准确，不可 Cherry-pick。❌ 「中国古代科学不行」→ ✅ 「中国古代科学发达，但没有发展出现代科学的原因是……」

19. **哲学/历史演变需要有清晰线索**：❌ 堆砌人名 → ✅ 「从泰勒斯自然哲学 → 苏格拉底『人』→ 柏拉图理念世界 → 亚里士多德体系」

20. **精简合并**：删重复内容，合并相似段落。段落不是越多越好。

21. **标点规范化**：中文标点规范使用（顿号分隔、破折号统一），引号用「」不用英文 ""。

22. **禁止排比三连**：不要三句相同句式并列（如「A戴着X的面纱，B戴着Y的面纱，C戴着Z的面纱」）。打破句式，每个用不同的动词和结构。

23. **不堆抽象名词**：不要一口气列出四五个抽象概念再加一句总结（如「虚荣、恐惧、占有欲、孤独。这些东西比爱情真实得多」）。挑一两个最有感觉的，用更口语的方式说。

24. **不用格言式短句收束**：不要用四字口号总结段落（如「环境塑造人」）。直接说事，别提炼金句。

25. **结尾不要三连问+升华**：不要用排比反问收尾再加一句「好的X都不Y」式金句。结尾用个人化表达，比如「我觉得他自己也不知道」比「好的小说都不给答案」好。

26. **不逐一列举做总结**：不要对每个人物/观点按相同格式罗列（如逐个说「A的爱是……B的爱是……C的爱是……」）。聚焦最想说的一两个，其他带过。

27. **避免书面化分析用语**：不要用文学评论课的词（如「完成人物塑造」「叙事策略」）。用口语替代：「写人」「讲故事的方式」。

28. **关联其他作品要节制**：对比其他书/电影必须确实有启发，不是为了显得有阅读量。如果对比不是特别必要，删掉比留着好。

29. **「让他A，也让他B」式对仗要拆开**：因果对称句太工整会有AI味。拆成两句，用转折或递进连接。❌「这种性格让他在疫区冷静面对死亡，也让他在私人关系里冷酷报复」→ ✅「这种性格在疫区是优点——冷静、不慌。但放到私人关系里，就变成了最冷酷的报复。」

## 禁止事项

- 不做流水账式的情节复述
- 不堆砌摘抄原文
- 不写空洞的"这本书很好，推荐大家读"类结语
- 不故弄玄虚、掉书袋
- 不人云亦云，没有自己的观点
- 不写元叙述（「这是我的观察」「这里我比较确定」等）
- 不居高临下（避免「赢家」「边缘化」等词）
